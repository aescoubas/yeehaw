# Yeehaw Codebase Evaluation Report

**Date:** 2026-03-03
**Scope:** Full codebase review — 69 source files, 23 test files, 232 passing tests, 84% coverage

---

## Summary

Yeehaw is a well-structured Python 3.12 multi-agent coding orchestrator with clean
module boundaries, consistent use of frozen dataclasses and type hints, and a solid
test suite. The architecture documentation is thorough (12 design docs), and the
signal protocol design is robust. Below are the gaps and inconsistencies found,
organized by severity.

---

## Critical

### 1. Stale auto-memory file

`~/.claude/projects/.../memory/MEMORY.md` describes a Go rewrite (`feat/refactor`
branch, Go 1.22, Cobra/BubbleTea) that no longer exists. The project is 100% Python.
This will mislead any future Claude session working on this project.

**Recommendation:** Rewrite MEMORY.md to reflect the current Python codebase.

### 2. `engine.py` is a 2,380-line monolith

`src/yeehaw/orchestrator/engine.py` handles signal processing, task dispatch, token
budget monitoring, merge/rebase orchestration, phase completion, worktree lifecycle,
and notification dispatch all in one class with 40+ methods. This is the highest-risk
file in the project and the hardest to reason about for correctness.

**Recommendation:** Decompose into focused modules:
- `TaskDispatcher` — queuing and launch
- `SignalProcessor` — signal file handling and state transitions
- `BudgetMonitor` — token and runtime budget enforcement
- `MergeOrchestrator` — rebase, conflict resolution, integration branches
- `PhaseManager` — phase advancement and verification

### 3. Duplicated token-parsing regex patterns

`TOTAL_TOKEN_PATTERNS`, `INPUT_TOKEN_PATTERNS`, `OUTPUT_TOKEN_PATTERNS`, and the
`_last_pattern_value` helper are copy-pasted between two files:

- `src/yeehaw/orchestrator/engine.py` (lines 70–96)
- `src/yeehaw/cli/status.py` (lines 46–72)

If a new agent output format is added, both must be updated independently — a
guaranteed drift bug.

**Recommendation:** Extract to a shared module (e.g., `src/yeehaw/tokens.py` or
`src/yeehaw/util/token_parser.py`).

### 4. Duplicated `_task_repo_root` helper

Three independent implementations with slightly different signatures:

| Location | Signature |
|---|---|
| `cli/stop.py:49` | `_task_repo_root(task, db_path) -> Path` |
| `cli/status.py:89` | `_task_repo_root(task, db_path) -> Path` |
| `cli/policy.py:296` | `_task_repo_root(task) -> Path` |

Any fix to repo-root resolution must be applied in three places.

**Recommendation:** Extract to a shared helper in `src/yeehaw/cli/` or a common
utilities module.

## High

### 5. No `TypedDict` or dataclass for task rows

Tasks flow through the entire system as `dict[str, Any]`. Every consumer
independently accesses keys like `task["id"]`, `task["status"]`, `task["phase_id"]`,
`task["tokens_used"]` with no compile-time or IDE safety. A missing or renamed column
silently produces `KeyError` at runtime.

**Recommendation:** Define a `TaskRow` TypedDict in `src/yeehaw/store/` and use it as
the return type for all Store task queries.

### 6. Shell injection risk in `tmux/session.py:pipe_output()`

```python
f"cat >> {log_path}"
```

The `log_path` is passed to tmux's `pipe-pane` which interprets it in a shell
context. Paths with spaces, quotes, or special characters will break or produce
unintended behavior.

**Recommendation:** Use `shlex.quote(str(log_path))`.

### 7. Heredoc delimiter collision in `agent/launcher.py`

The task prompt is wrapped in `<<'YEEHAW_PROMPT_EOF'` (line 143) but the prompt
content is never validated. A task description containing the literal string
`YEEHAW_PROMPT_EOF` would terminate the heredoc early and produce a broken launcher
script.

**Recommendation:** Either sanitize the prompt to escape/remove the delimiter, or use
a dynamically generated unique delimiter.

### 8. `merge_resolver.py` whitespace-conflict edge case (lines 211–212)

```python
if not ours_normalized and not theirs_normalized:
    return False
```

When both sides of a hunk normalize to empty strings, the conflict is purely
whitespace — but the code returns `False` (not whitespace-only), rejecting
auto-resolution. Deletion-vs-deletion conflicts that differ only in trailing
whitespace should be auto-resolvable.

**Recommendation:** Return `True` (or `continue` to the next hunk) when both sides
normalize to empty.

### 9. Subprocess calls without timeouts in `git/worktree.py`

`subprocess.run(["git", ...], check=True)` is called with no `timeout` parameter. On
a network-backed repo or a stalled index lock, these hang indefinitely. The
orchestrator's main loop blocks until the subprocess returns.

**Recommendation:** Add `timeout=60` (or a configurable value) to all subprocess
calls in the git module.

## Medium

### 10. Inconsistent error-handling patterns across modules

| Module | Pattern |
|---|---|
| `config/loader.py` | Raises `ValueError` with detailed context |
| `hooks/runner.py` | Typed exception hierarchy (`HookError`, `HookTimeoutError`) |
| `notify/webhook.py` | Returns success/failure booleans |
| `cli/stop.py` | Silent failure on cleanup |
| `orchestrator/engine.py` | Broad `except Exception` in 20+ places |

No project-wide error strategy. Some failures log, some raise, some silently
continue.

**Recommendation:** Document an error-handling convention. Consider a base
`YeehawError` exception class and module-specific subclasses.

### 11. Coverage gaps in key modules

The project target is 80%+. Six modules fall below that:

| Module | Coverage | Notes |
|---|---|---|
| `orchestrator/engine.py` | 76% | Lowest coverage for highest-risk file |
| `policy/models.py` | 71% | Policy rule dataclass validation |
| `context/models.py` | 71% | Memory pack validation |
| `config/loader.py` | 72% | Feature flag loading edge cases |
| `scm/github.py` | 72% | GitHub adapter (likely no integration tests) |
| `planner/generate.py` | 74% | Roadmap generation workflow |

**Recommendation:** Prioritize `engine.py` and `config/loader.py` for additional
test coverage since they are on the critical path.

### 12. `args: Any` throughout CLI handlers

Every CLI handler (`run.py`, `status.py`, `stop.py`, etc.) types its first parameter
as `args: Any` instead of `argparse.Namespace`. This defeats type checking for
attribute access on args.

**Recommendation:** Use `argparse.Namespace` or define per-command TypedDicts.

### 13. No CI/CD pipeline or meaningful lint step

No type checker (mypy/pyright), no linter (ruff/flake8). The Makefile lint target only runs
`py_compile` on a single file:

```makefile
lint:
    uv run python -m py_compile src/yeehaw/cli/main.py
```

This catches syntax errors in one file only.

**Recommendation:** Add at minimum:
- `ruff check src/ tests/` for linting
- `mypy src/` or `pyright` for type checking

### 14. MCP server bypasses Store API

`mcp/server.py:409–414` directly accesses `store._conn.execute()` instead of going
through the Store public interface:

```python
store._conn.execute(
    "UPDATE tasks SET assigned_agent = ?, updated_at = ? WHERE id = ?",
    (assigned_agent, store._now(), task_id),
)
```

This couples the MCP server to SQLite internals and bypasses any future validation or
event hooks in the Store layer.

**Recommendation:** Add a `store.update_task_agent(task_id, agent)` method.

## Low

### 15. `FeatureFlags` / `FEATURE_FLAG_NAMES` manual sync

`config/models.py` defines both a `FEATURE_FLAG_NAMES` tuple (lines 7–16) and a
`FeatureFlags` frozen dataclass. Adding a feature flag requires updating both. No
assertion or code generation enforces consistency.

**Recommendation:** Derive `FEATURE_FLAG_NAMES` from the dataclass fields:
```python
FEATURE_FLAG_NAMES = tuple(f.name for f in fields(FeatureFlags))
```

### 16. No agent binary availability check

`agent/profiles.py` defines agent commands (`"claude"`, `"gemini"`, `"codex exec
..."`) but never verifies the binary exists in `$PATH`. Discovery only happens at
task dispatch time, wasting a worktree + tmux session before failing.

**Recommendation:** Add a `profile.is_available() -> bool` check using
`shutil.which()`, and call it before dispatch.

### 17. `runtime_config.py` silently swallows TOML parse errors

```python
except (tomllib.TOMLDecodeError, OSError):
    return []
```

If the codex TOML config is malformed, MCP server names silently default to empty —
no log, no warning.

**Recommendation:** Log a warning when TOML parsing fails.

### 18. No SQLite connection health check in Store

The SQLite connection is opened once and assumed alive for the entire orchestrator
lifetime. No ping or reconnect logic if the DB file is moved, locked, or corrupted
mid-run.

**Recommendation:** Add a lightweight health check (e.g., `SELECT 1`) before critical
write paths, or use a connection factory that re-opens on failure.

### 19. Uncommitted working tree changes

The current working tree has a coherent set of changes (auto-queue pending tasks,
token budget reconciliation, phase verification refactor) across 10 files with
matching tests — all 232 tests passing. These should be committed to avoid accidental
loss.

## Architectural Observations

### Strengths

- Clean module boundaries with single-responsibility packages
- Consistent use of frozen dataclasses and type hints throughout
- Solid test suite: 232 tests, 84% coverage, fast execution (11s)
- Well-documented architecture (12 design docs in `ARCHITECTURE/`)
- Signal protocol design is robust: watchdog + debounce + polling fallback + parse retries
- Schema migration system handles legacy upgrades gracefully with backup
- Good use of `pathlib.Path` throughout (no raw string paths)
- WAL mode + busy timeout for SQLite concurrency

### Structural Debt

- `engine.py` should be decomposed into 5 focused modules (see Critical #2)
- Shared utilities (token parsing, task repo root, ANSI stripping) need a common home
- Task rows need a `TypedDict` to replace `dict[str, Any]` across the codebase
- A project-wide error convention would prevent the inconsistent mix of exceptions,
  booleans, and silent failures
- The `_slugify()` and `_dedupe_preserve_order()` helpers are duplicated across
  `policy/loader.py` and `context/loader.py`
